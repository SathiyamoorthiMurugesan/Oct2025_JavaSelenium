How to import a project from repository to your local(your system) - using eclipse alone 

1. You should be having access to the repository(repo) you are going to clone.
2. Ask the repo owner to add you as a collabarator to that repo. 
3. Go the the repo in github
4. Click on code and copy the ssh path 
5. Open Eclipse 
6. File -> import or in package Explorer, right click on the empty space and click on import.
7. Select Git -> Project from git -> next 
8. Select repo source -> clone url
9. Source git repo - paste the URI (usally it will be pasted automatically, if you have copied the ssh url then no change are needed) -> next
10. Branch selection - choose what the branches you want -> next
11. Local Destination - path will come automatically(recommended). if you want to change you can change otherwise dont -> next
(After the above step, in your local, you can see the project.)
12. Select Wizard -> choose import existing eclipse project

10. File -> import -> select maven - existing maven project ->next

13. Now you can see the project in your eclipse

JVM - Java Virtual Machine 
* This can not be seen
* it starts the execution(creates Runtime Environment)
* It first checks for main method and executes the codes inside the main method
* It creates the code flow, in java the flow is called "Thread"  - "main Thread"
* JVM provides run time environment in which Java byte codes can be executed
* loads code| verifies code| execute code| provides run time environment

JRE - Java Runtime Environment
* Physically exist
* Contains JVM and set of libraries
* JVM uses these libraries to execute the codes

JDK - Java Developement kit
* It contains JVM, set of libraries and dev tools (JRE + dev tools)
* Used to develop and execute the codes
* Dev tools + debugger + compiler + java doc

public class ExcelReader {
    public static void main(String[] args) {
        String excelFilePath = "example.xlsx"; // Path to your Excel file
        try (FileInputStream fis = new FileInputStream(new File(excelFilePath));
             XSSFWorkbook workbook = new XSSFWorkbook(fis)) {

            XSSFSheet sheet = workbook.getSheetAt(0); // Get the first sheet

            for (Row row : sheet) { // Iterate through each row
                for (Cell cell : row) { // Iterate through each cell in the row
                    switch (cell.getCellType()) {
                        case STRING:
                            System.out.print(cell.getStringCellValue() + "\t");
                            break;
                        case NUMERIC:
                            System.out.print(cell.getNumericCellValue() + "\t");
                            break;
                        case BOOLEAN:
                            System.out.print(cell.getBooleanCellValue() + "\t");
                            break;
                        case FORMULA:
                            System.out.print(cell.getCellFormula() + "\t");
                            break;
                        default:
                            System.out.print("Unknown Cell Type\t");
                    }
                }
                System.out.println(); // Move to the next line for each row
            }

        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}

Casting in Java

Casting in Java refers to converting one data type into another. There are two main types of casting:

1. Primitive Type Casting
	a. Widening Casting (Implicit)
		Automatically done by Java.
		Smaller to larger type conversion.

Examples:
	int a = 10;
	double b = a;  // int to double
	Order:
	byte â†’ short â†’ int â†’ long â†’ float â†’ double

	b. Narrowing Casting (Explicit)
		Manually done using parentheses.
		Larger to smaller type conversion.

Examples:
	double a = 10.5;
	int b = (int) a;  // double to int, value becomes 10


2. Reference Type Casting (Objects)
	Used when dealing with class hierarchies.

	a. Upcasting (Implicit)
		Subclass to superclass.
		Safe and automatic.
Example:
	class Animal {}
	class Dog extends Animal {}
	Animal a = new Dog();  // Upcasting
	
	b. Downcasting (Explicit)
		Superclass to subclass.
		Needs explicit cast and may throw ClassCastException at runtime if types are incompatible.

Example:
	Animal a = new Dog();   // Upcasting
	Dog d = (Dog) a;        // Downcasting
	Important Notes
	Always check with instanceof before downcasting:

		if (a instanceof Dog) {
			Dog d = (Dog) a;
			}
Casting is different from conversion in high-level classes like String â†’ int, which requires parsing (e.g., Integer.parseInt("123")).

Wait in Selenium

In Selenium WebDriver, waits are used to synchronize the execution of test scripts with the web application under test. 
This ensures that Selenium waits for certain conditions before performing actions, 
helping to avoid flaky or failing tests due to timing issues (e.g., elements not being ready).

1. Implicit Wait
	Tells WebDriver to wait a specified amount of time when trying to find an element if it is not immediately available.
	Applies globally to all elements.

Example:
	driver.manage().timeouts().implicitlyWait(Duration.ofSeconds(10));
	
2. Explicit Wait
	Waits for a specific condition to occur before proceeding further in the code.
	More powerful and customizable than implicit waits.

Common Conditions (from ExpectedConditions):
	visibilityOfElementLocated
	elementToBeClickable
	presenceOfElementLocated
	alertIsPresent
	titleContains

Example:
	WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(10));
	WebElement element = wait.until(ExpectedConditions.visibilityOfElementLocated(By.id("username")));
	
3. Fluent Wait
	A more advanced form of explicit wait.
	Allows you to define:
		Polling frequency
		Exceptions to ignore
		Timeout duration

Example:
	Wait<WebDriver> wait = new FluentWait<>(driver)
    	.withTimeout(Duration.ofSeconds(30))
    	.pollingEvery(Duration.ofSeconds(5))
    	.ignoring(NoSuchElementException.class);

WebElement element = wait.until(driver -> driver.findElement(By.id("username")));

ðŸ§  When to Use Which?
	Situation	Recommended Wait
	Wait for element load globally	Implicit Wait
	Wait for specific condition	Explicit Wait
	Retry with custom polling and exception handling	Fluent Wait

âŒ Avoid This: Thread.sleep()
	Hard-coded wait.
	Always waits full duration, even if condition is met earlier.
	Slows down your test unnecessarily.

âœ… Best Practices
	Use explicit waits for dynamic elements.
	Avoid mixing implicit and explicit waits.
	Use FluentWait when polling is required (e.g., waiting for a spinner to disappear).

TestNG assertion

Assertions in Selenium UI testing (with TestNG or any test framework) are essential 
because they validate that the application behaves as expected during or after an interaction. 
Simply put, assertions are what turn your automation scripts into actual tests.

Assertions immediately fail the test if something goes wrong, allowing early detection of bugs.

Without assertions, your Selenium script may:

Click buttons
Fill out fields
Navigate pages

But it doesn't verify if those actions had the correct result.

* Test with assertions = proves the system works
* Test without assertions = just simulates actions

Common TestNG Assertions
	Method	Description
		Assert.assertEquals(actual, expected)	Checks if two values are equal
		Assert.assertTrue(condition)	Validates that a condition is true
		Assert.assertFalse(condition)	Validates that a condition is false
		Assert.assertNotNull(object)	Ensures the object is not null
		Assert.assertNull(object)	Ensures the object is null
		
Example: With vs Without Assertion
Action					Without Assertion				With Assertion
Go to Login Page		Opens the page				Confirms page loaded correctly
Enter credentials		Types values				Validates fields accepted input
Click login				Clicks the button			Confirms dashboard appears
Test outcome			No validation				Fails if title != "Dashboard"

When there is a risk in completing a story
	E.g: Developer gives story to testing when there is only 3 days left. But in order to complete the testing you need 7 days
In this case
	1. No other choice you need to complete the testing in 3 days itself.
	2. Complete as much as possible, then create a new QA(Testing) story in next sprint
	3. Move the entire story from this sprint to next sprint
	4. Let your scrum master to ask for extension of the current sprint

Sprint ceremonies

PI Planning
	* Client or client's representative explains their initiative or goal or tells the feature they want in their app
	* Product owner, Lead, Scrum master and Tech arch from each agile team will participate
	* Each agile team will be give one or more feature to work on.

* In order to start the sprint, I should know what stories my team need to work before the start of the sprint

How many stories we are going to work
Who is going to work on which story
What the priority
Who is on leave during the sprint/ Any Holidays

Sprint planning - Sprint start 
 * Mostly starts by Wednesday or Thursday
 * Normally two weeks
 * Always happen during the previous print
 * What are the stories are we going to work on this sprint
 * Who is going to work on which story
 * How much story point for each story
 * How many story points we are going to work on this sprint
 
 * Sprint ends on the last day of the Sprint
 
Sprint refinement or backlog refinement
	* Product owner or Dev lead or tech arch, explains each stories in the backlog
	* What is the expectation
	* Defines Acceptance criteria
	* What technologies involved in the story
	* Testing needed or not 

Status Call/Huddle

* What you did yesterday
* Whats your plan for today
* What will you do on tomorrow. 
* Any road blocks you faced/will face to complete the work

Run by - Scrum master
Max time for the meeting - 15 minutes
No technical talks

* Sprint Demo
	* Where all the agile team members participate
	* We need to demonstrate what we worked 
	* Either we need to give demo or show the screenshots(for the time being)
	* Usually given by Product Owner or Lead 
	* Sometimes, tester also need to give

* Sprint Retrospective
	During the last sprint
		* What went well
		* What did not go well
		* What we need to continue doing
		* Appreciation
		* Dependencies from other team went well or not
